= Spring AI Integration
:navtitle: Spring AI

Vectors provides a Spring AI `VectorStore` adapter and a Spring Boot starter. `JavaVectorsVectorStore` wraps a `VectorCollection` and an `EmbeddingModel`, implementing Spring AI's `VectorStore` interface as a drop-in, indexed replacement for `SimpleVectorStore`.

NOTE: Vectors requires JDK 25 with the Vector API enabled: run with `--add-modules jdk.incubator.vector`.

The adapter's Spring AI dependency is `compileOnly` and it binds only to the stable `VectorStore` SPI, so one artifact works across Spring AI **1.0.x, 1.1.x, and 2.0.x** — you bring your own Spring AI version. CI compiles and runs the adapter's tests against each line on every change.

== Dependency Setup

=== Direct Adapter

Add `vectors-spring-ai` to use `JavaVectorsVectorStore` directly:

[tabs]
====
Gradle::
+
[source,kotlin,subs="attributes+"]
----
dependencies {
    implementation("com.integrallis:vectors-spring-ai:{vectors-version}")
    implementation("org.springframework.ai:spring-ai-vector-store:1.1.4")
}
----

Maven::
+
[source,xml,subs="attributes+"]
----
<dependency>
    <groupId>com.integrallis</groupId>
    <artifactId>vectors-spring-ai</artifactId>
    <version>{vectors-version}</version>
</dependency>
----
====

=== Spring Boot Starter

== Using JavaVectorsVectorStore

Build a `VectorCollection`, then pass it together with an `EmbeddingModel` to
`JavaVectorsVectorStore.builder(embeddingModel, collection)`:

[source,java]
----
VectorCollection collection = VectorCollection.builder()
    .dimension(embeddingModel.dimensions())
    .metric(SimilarityFunction.COSINE)
    .indexType(IndexType.HNSW)
    .build();

VectorStore vectorStore = JavaVectorsVectorStore.builder(embeddingModel, collection)
    .collectionName("my-docs")   // used in observation metadata; default "default"
    .commitAfterAdd(true)        // commit after each add() so writes are searchable; default true
    .build();

// Standard Spring AI API
vectorStore.add(List.of(
    new Document("hello world"),
    new Document("goodbye world")));

List<Document> results = vectorStore.similaritySearch(
    SearchRequest.builder().query("hello").topK(5).build());
----

The builder embeds each `Document` through the supplied `EmbeddingModel` before staging it in the
collection, and validates at build time that the model's dimension matches the collection's.
`similaritySearch` embeds the query, translates any filter expression, and maps hits back to Spring
AI `Document` instances with their similarity `score` populated.

=== Diversity re-ranking (MMR)

RAG retrieval often returns several near-duplicate chunks that waste the LLM context window. Enable
**Maximal Marginal Relevance** on the store to trade a little relevance for diversity — no second
datastore, no external rerank API:

[source,java]
----
VectorStore vectorStore = JavaVectorsVectorStore.builder(embeddingModel, collection)
    .mmr(0.6f)   // lambda: 1.0 = pure relevance, 0.0 = max diversity; typical 0.5-0.7
    .build();

// Every similaritySearch now over-fetches topK x 4 candidates and re-ranks them down to
// topK diverse-but-relevant results using the collection's metric.
List<Document> results = vectorStore.similaritySearch(
    SearchRequest.builder().query("hello").topK(5).build());
----

Use `.mmr(lambda, fetchMultiplier)` to control the candidate-pool size (default multiplier `4`).
MMR is computed server-side over the over-fetched pool via the fused-SIMD kernels — it runs on a
small candidate set, not the whole corpus, so the overhead is sub-millisecond for typical `topK`.

== Spring Boot Properties

When using the starter, configure via `application.yml`:

[source,yaml]
----
java-vectors:
  dimension: 1536
  metric: COSINE
  index-type: HNSW
  quantizer: SQ8
  storage-path: /var/lib/vectors/my-collection
  auto-commit-threshold: 1
  hnsw:
    m: 16
    ef-construction: 200
  vamana:
    max-degree: 64
    search-list-size: 128
    alpha: 1.2
----

The starter auto-configures:

* A `VectorCollection` bean from the properties above.
* A `JavaVectorsVectorStore` bean (conditional on an `EmbeddingModel` bean being present).

Both beans are `@ConditionalOnMissingBean`, so you can override them with custom `@Bean` definitions.

== Filter Translation

Spring AI filter expressions are translated to the java-vectors `Filter` AST.
All standard operators are supported:

[cols="1,1", options="header"]
|===
| Spring AI | java-vectors

| `EQ`   | `Filters.eq()`
| `NE`   | `Filters.not(Filters.eq())`
| `GT`   | `Filters.gt()`
| `GTE`  | `Filters.gte()`
| `LT`   | `Filters.lt()`
| `LTE`  | `Filters.lte()`
| `IN`   | `Filters.inStr()` / `Filters.inNum()`
| `NIN`  | `Filters.not(Filters.inStr())`
| `AND`  | `Filters.and()`
| `OR`   | `Filters.or()`
| `NOT`  | `Filters.not()`
|===

== Embedding Cache

Combine with `vectors-cache-spring-ai` to cache embedding model calls:

[source,java]
----
VectorCache<String, float[]> cache = CaffeineVectorCache.<String, float[]>builder()
    .maximumSize(10_000)
    .build();

EmbeddingModel cached = new CachingEmbeddingModel(embeddingModel, cache);
----

The caching decorator intercepts `embed()` and `embedAll()` calls, serving cached results for previously seen text inputs and batching only the cache misses into a single delegate call.
