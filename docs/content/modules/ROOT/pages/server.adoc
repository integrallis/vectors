= HTTP Server
:navtitle: HTTP Server

The `vectors-server` module exposes a `VectorCollection` over a JSON REST API.
It is built on Helidon SE 4 (Nima), which serves each request on a virtual thread.

`vectors-server` is part of the xref:distributed.adoc[distributed tier].

Like the rest of the project, the server runs on JDK 25 with `--add-modules jdk.incubator.vector`. The Gradle `run` task and the application launcher apply this flag automatically; add it yourself when launching the jar directly.

== Starting the Server

[source,bash]
----
cd vectors
./gradlew :vectors-server:run
----

The server listens on port 8287 by default.

=== Configuration

The launcher parses CLI options with picocli:

[source,bash]
----
java --add-modules jdk.incubator.vector -jar vectors-server.jar \
  --port 9090 \
  --data-dir /var/lib/vectors \
  --max-connections 512 \
  --shutdown-timeout-seconds 30 \
  --api-key "$VECTORS_API_KEY"
----

[cols="2,3", options="header"]
|===
| Option | Description

| `--port`                     | Listen port (default 8287)
| `--data-dir`                 | Directory for persistent collections; omit for in-memory
| `--max-connections`          | Maximum concurrent connections (default 512)
| `--shutdown-timeout-seconds` | Graceful shutdown grace period (default 30)
| `--api-key`                  | Bearer token for authentication; falls back to the `VECTORS_API_KEY` environment variable
|===

When an API key is configured, every route requires an `Authorization: Bearer <token>` header.

Or configure the server programmatically:

[source,java]
----
ServerConfig config = ServerConfig.withPort(9090)
    .withDataDir(Path.of("/var/lib/vectors"));
ServerHandle handle = VectorsServer.start(config);
----

== REST API

All routes are versioned under `/v1`.

=== Observability

[cols="1,3,2", options="header"]
|===
| Method | Path | Description

| `GET` | `/v1/health`                     | Liveness check; returns `{"status":"UP","collections":N}`
| `GET` | `/v1/readyz`                     | Readiness check
| `GET` | `/v1/metrics`                    | Prometheus text-format metrics
| `GET` | `/v1/collections/\{name\}/epoch` | Current collection epoch; supports `ETag` / `If-None-Match` (304)
|===

The server has no HTTP shutdown route. Shutdown is driven by a JVM shutdown hook and the configured grace period; `ServerHandle.stop()` stops an embedded instance.

=== Collections

[cols="1,3,2", options="header"]
|===
| Method | Path | Description

| `POST`   | `/v1/collections`         | Create a new collection (201)
| `GET`    | `/v1/collections`         | List all collections
| `GET`    | `/v1/collections/\{name\}`  | Describe a collection
| `DELETE` | `/v1/collections/\{name\}`  | Drop a collection (204)
|===

==== Create a Collection

`name`, `dimension`, `metric`, and `indexType` are required.
`metric` is one of `EUCLIDEAN`, `DOT_PRODUCT`, `COSINE`, `MAXIMUM_INNER_PRODUCT`.
`indexType` is one of `FLAT`, `HNSW`, `VAMANA`, `IVF_FLAT`, `IVF_PQ`.
The optional `quantizer` and index-specific tuning fields (`hnswM`, `hnswEfConstruction`, and the `vamana*` / `ivf*` families) are accepted as needed.

[source,bash]
----
curl -X POST http://localhost:8287/v1/collections \
  -H "Content-Type: application/json" \
  -d '{
    "name": "my-collection",
    "dimension": 384,
    "metric": "COSINE",
    "indexType": "HNSW",
    "quantizer": "SQ8",
    "hnswM": 16,
    "hnswEfConstruction": 200
  }'
----

=== Documents

[cols="1,3,2", options="header"]
|===
| Method | Path | Description

| `POST`   | `/v1/collections/\{name\}/documents`      | Upsert a batch of documents
| `GET`    | `/v1/collections/\{name\}/documents/\{id\}` | Get a document by ID
| `DELETE` | `/v1/collections/\{name\}/documents/\{id\}` | Delete a document (204)
| `POST`   | `/v1/collections/\{name\}/commit`         | Commit staged writes (204)
|===

==== Upsert Documents

Each document carries an `id` and `vector`, with optional `text`, `metadata`, and Base64 `blob` fields.
By default an upsert commits immediately; pass `?commit=false` to stage the batch and commit it later with the commit endpoint.

[source,bash]
----
curl -X POST http://localhost:8287/v1/collections/my-collection/documents \
  -H "Content-Type: application/json" \
  -d '{
    "documents": [
      {
        "id": "doc-1",
        "vector": [0.1, 0.2, 0.3],
        "text": "hello world",
        "metadata": {"author": "alice", "year": 2024}
      }
    ]
  }'
----

=== Search

[cols="1,3,2", options="header"]
|===
| Method | Path | Description

| `POST` | `/v1/collections/\{name\}/search`  | Search for nearest neighbors
|===

==== Search Example

The query is `queryVector`; `k` sets the number of results.
An optional `filter` restricts results by metadata: a field predicate is `{"field": ..., "<op>": ...}` where `<op>` is one of `eq`, `ne`, `in`, `nin`, `gt`, `gte`, `lt`, `lte`, and predicates compose under `and`, `or`, and `not`.

[source,bash]
----
curl -X POST http://localhost:8287/v1/collections/my-collection/search \
  -H "Content-Type: application/json" \
  -d '{
    "queryVector": [0.1, 0.2, 0.3],
    "k": 10,
    "filter": {
      "and": [
        {"field": "author", "eq": "alice"},
        {"field": "year", "gte": 2023}
      ]
    }
  }'
----

Response hits include the document `id` and a similarity `score` (higher is closer).
Set `includeVector`, `includeText`, or `includeMetadata` to control which document fields are returned.

=== Bulk and Blobs

[cols="1,3,2", options="header"]
|===
| Method | Path | Description

| `GET`  | `/v1/collections/\{name\}/documents`       | List documents (paged)
| `POST` | `/v1/collections/\{name\}/vectors-batch`   | Fetch vectors for a set of IDs
| `GET`  | `/v1/collections/\{name\}/sample`          | Sample documents from the collection
| `GET`  | `/v1/collections/\{name\}/blobs/\{docId\}`   | Fetch a document's raw blob
|===

=== Events

[cols="1,3,2", options="header"]
|===
| Method | Path | Description

| `GET` | `/v1/events` | SSE stream of collection epoch changes
|===

The SSE endpoint opens with a `hello` event and then emits an `epoch` event whenever a collection's generation advances, letting clients invalidate caches on ingest, delete, or commit.

== Persistence

Started with a `--data-dir`, the server persists collections to disk.
On startup, `CollectionDiscovery` scans the data directory and reopens every collection that has a valid `CURRENT` pointer and generation.

Collections created without a data directory are in-memory and released on shutdown.

== Embedded Usage

Start the server in-process for tests. `ServerConfig.forTesting()` binds an ephemeral port and runs in-memory:

[source,java]
----
ServerHandle handle = VectorsServer.start(ServerConfig.forTesting());
int port = handle.port();
// ... run tests against http://localhost:<port> ...
handle.stop();
----

== Java Client

`vectors-server-client` is a typed Java client for the REST API, built on the JDK `HttpClient` with
Jackson — no extra HTTP dependency. `VectorsServerClient` is thread-safe and `AutoCloseable`, and
maps 4xx/5xx responses to a `VectorsServerException`.

NOTE: `vectors-server-client` is source-available in the repository and not yet published to Maven Central.

[source,java]
----
try (VectorsServerClient client = new VectorsServerClient("http://localhost:8287")) {
  client.createCollection("demo", 384, "COSINE", "HNSW", Map.of());
  client.upsertDocuments("demo", List.of(
      new DocumentPayload("a", vectorA, "alpha", Map.of(), null)));
  client.commit("demo");

  List<SearchHit> hits = client.search("demo", queryVector, 10, null, Map.of());
  // hybrid (server-side RRF fusion of vector + text):
  List<SearchHit> hybrid = client.hybridSearch("demo", queryVector, "alpha", 10, "RRF");
}
----

The client also covers collection lifecycle (`listCollections`, `describe`, `deleteCollection`),
bulk reads (`previewDocuments`, `vectorsBatch`, `sample`), blob retrieval (`getBlob`), and a
`isHealthy()` probe. Pass an API key to the constructor for bearer-token auth.
