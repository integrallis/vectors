= Sizing Guide
:navtitle: Sizing Guide

This page helps you estimate how many vectors fit in a given memory budget and pick an
index / quantizer combination that matches your dataset and SLO.

== Memory model

Vectors uses two memory pools:

* **Heap** -- Java objects, HNSW/Vamana graph nodes, in-memory vector arrays
  (`float[][]`). Bounded by `-Xmx`.
* **Off-heap (`MemorySegment`)** -- memory-mapped `vectors.bin` and `graph.bin` files used
  by persistent collections. Bounded by RAM plus OS page-cache behaviour, not by `-Xmx`.

All calculations below assume the **in-memory** (heap-resident) configuration unless
noted. Persistent/mmap-backed collections lift the per-vector heap cost to roughly a few
dozen bytes per node (graph adjacency) and push the float payload off-heap.

== Per-vector cost by encoding

The dominant cost is the vector payload. For dimension `dim`:

[cols="2,2,3", options="header"]
|===
| Encoding | Bytes per vector | Notes
| `FLOAT32`        | `4 Ã— dim`       | Full precision; reference.
| `INT8` (scalar)  | `1 Ã— dim` + 8   | 1 byte per element + per-vector `{min, scale}`.
| `BINARY` (BQ)    | `dim / 8`       | 1 bit per element, rounded up to the nearest byte.
| `PQ` (M subspaces, Ks=256) | `M` | 1 byte per subspace; codebook is amortised across the dataset.
|===

Product quantization stores one 8-bit code per subspace (`M` bytes per vector) plus a
shared codebook of `M Ã— Ks Ã— (dim / M) Ã— 4` bytes -- typically a few MB total and
amortised across the entire corpus. The codebook dominates only at very small `n`.

For rescore-style configurations (quantized navigator + full-precision retained for exact
rescore via `IndexRefine`), add the `FLOAT32` cost on top of the navigator encoding.

the default `M = dim / 8` is used. Add `4 Ã— dim` per vector if you retain full-precision

== Tuning knobs

* **HNSW `M`** -- higher `M` increases recall but raises per-node overhead by `~12 Ã— M`
  bytes. Typical range: `16â€“48`.
* **Vamana `R`** -- graph degree; similar effect to HNSW `M`. Typical range: `32â€“64`.
* **`overQueryFactor`** -- coarse-pass over-fetch; recall tuning without rebuild.
* **`searchMultiStart`** -- intra-query parallelism for HNSW via virtual threads; raises
  QPS at high recall at the cost of extra CPU per query.
* **IVF `nprobe`** -- number of clusters scanned; controls recall/latency trade.
* **PQ `M`** -- number of subspaces. Compression ratio is `4 Ã— dim / M`; recall drops as
  `M` shrinks. Defaults to `max(1, dim / 8)`. Typical range: `dim / 16` to `dim / 4`.
* **PQ `rescoreFactor`** -- wide-heap multiplier used when full-precision vectors are
  retained. `1` = pure ADC; `4â€“10` typically closes the recall gap to the flat baseline.

== Off-heap (mmap) note

For persistent collections the `vectors.bin` payload is zero-copy via `MemorySegment`
over mmap. Heap pressure drops to the graph adjacency lists plus per-document metadata;
the OS page cache is the effective capacity ceiling. See the Getting Started guide for
enabling `PersistentVectorCollection`.
