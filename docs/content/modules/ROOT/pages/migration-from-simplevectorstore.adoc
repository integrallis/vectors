= Migrating from Spring AI SimpleVectorStore
:navtitle: Migrating from SimpleVectorStore

`SimpleVectorStore` is the in-memory reference implementation of Spring AI's
`VectorStore`. It holds every vector in a `ConcurrentHashMap` and scans the full map on
every query. There are two ways to replace it with `vectors`, depending on whether you
want to keep programming against Spring AI's `VectorStore` interface.

== Option 1: Drop-in replacement (keep Spring AI)

If your code calls the Spring AI `VectorStore` API (`add`, `similaritySearch`,
`delete`), swap `SimpleVectorStore` for `JavaVectorsVectorStore`. It is a Spring AI
`VectorStore` backed by an indexed, persistent `VectorCollection`, so **only the
construction changes â€” every call site stays the same.**

[source,java]
----
// Before -- Spring AI SimpleVectorStore
VectorStore store = SimpleVectorStore.builder(embeddingModel).build();

// After -- Vectors, still a Spring AI VectorStore
VectorCollection collection = VectorCollection.builder()
    .dimension(embeddingModel.dimensions())
    .indexType(IndexType.HNSW)
    .metric(SimilarityFunction.COSINE)
    .build();
VectorStore store = JavaVectorsVectorStore.builder(embeddingModel, collection).build();
----

Everything downstream is untouched â€” the embedding model still runs inside the store, and
queries, filters, and writes use the same Spring AI types:

[source,java]
----
// Unchanged: adds, queries, and filter expressions are the Spring AI VectorStore API
store.add(List.of(new Document("hello"), new Document("world")));

List<Document> hits = store.similaritySearch(
    SearchRequest.builder().query("hello").topK(5).build());

List<Document> filtered = store.similaritySearch(
    SearchRequest.builder().query("q").topK(5).filterExpression("author == 'alice'").build());
----

To persist, pass an absolute `storagePath` to the `VectorCollection` builder; nothing else
changes. See xref:spring-ai.adoc[Spring AI adapter] for the auto-configuration and Spring
Boot starter.

== Option 2: Program directly against VectorCollection

If you want to drop the Spring AI dependency, use `VectorCollection` directly. You then own
the embedding call, and in return you get the native `SearchResult` API, batch search, a
typed filter tree, and control over the write/commit lifecycle.

Replace the store construction:

[source,java]
----
// Before -- Spring AI SimpleVectorStore
VectorStore store = SimpleVectorStore.builder(embeddingModel).build();

// After -- native Vectors collection
VectorCollection store = VectorCollection.builder()
    .dimension(embeddingModel.dimensions())
    .indexType(IndexType.HNSW)
    .metric(SimilarityFunction.COSINE)
    .build();
----

=== Writes

Spring AI's store embeds `Document` text for you. The native API takes the vector directly,
so you embed with your model (or reuse your existing pipeline):

[source,java]
----
store.addAll(List.of(
    Document.of("1", embeddingModel.embed("hello"), "hello"),
    Document.of("2", embeddingModel.embed("world"), "world")));
----

=== Queries

You embed the query, then read results from a `SearchResult`, whose `hits()` are ordered
`Hit` records carrying the `id`, similarity `score`, and backing `Document`:

[source,java]
----
float[] query = embeddingModel.embed("hello");
SearchResult result = store.search(SearchRequest.builder(query, 5).build());
for (SearchResult.Hit hit : result.hits()) {
    String id = hit.id();
    float score = hit.score();
    Document doc = hit.document();
}
----

=== Metadata filters

Spring AI expresses filters as an SpEL-like string. The native API uses a typed filter tree:

[source,java]
----
// Spring AI: SearchRequest.query("q").withFilterExpression("author == 'alice'")
SearchResult result = store.search(
    SearchRequest.builder(query, 5)
        .filter(Filters.eq("author", "alice"))
        .build());
----

`Filters` exposes `eq`, `gt`, `gte`, `lt`, `lte`, `inStr`, `inNum`, `between`, `and`, `or`,
`not`, and `all` (inequality is `Filters.not(Filters.eq(...))`).

=== Batched queries

`SimpleVectorStore` has no batching API â€” callers thread `similaritySearch` themselves.
The native API batches directly, with per-query parallelism on virtual threads:

[source,java]
----
List<SearchResult> results = store.searchBatch(List.of(
    SearchRequest.builder(embed("q1"), 5).build(),
    SearchRequest.builder(embed("q2"), 5).build(),
    SearchRequest.builder(embed("q3"), 5).build()));
----

== Persistence

`SimpleVectorStore` persists the map as JSON via `save(File)` / `load(File)`. Either
migration path gains segment-oriented mmap storage: pass an absolute `storagePath` to the
`VectorCollection` builder and the collection is durable instead of purely in-memory. See
xref:persistence.adoc[Persistence] for the full walkthrough.

== What you gain

* **Indexed approximate search** â€” HNSW graph navigation replaces the full linear scan.
  Throughput and recall depend on the corpus, hardware, and index parameters.
* **Intra-query parallelism** â€” `searchMultiStart` on `SearchRequest` fans out beam
  searches across virtual threads, trading CPU for latency at high recall.
* **Quantization** â€” set `QuantizerKind.SQ8` or `QuantizerKind.RABITQ` on the builder to
  shrink the encoded vector payload. Measure recall on the target corpus.
* **Composition** â€” wrap any backend with `IndexShards` (horizontal partitioning) or
  `IndexReplicas` (concurrency fan-out) without changing the call sites.

== What you keep

* The same `Document`-style abstraction (id + vector + optional text + metadata).
* The same "build once, query many" lifecycle.
* A pure-JVM, in-process deployment: no JNI library or GPU backend is required by the CPU
  artifacts. Persistent storage uses `posix_madvise` through the Foreign Function & Memory API.
