= The Distributed Vector Database
:navtitle: Distributed Database

The embedded engine documented in the rest of this guide is a **free, Apache-licensed library**: it
runs inside your application process and stores data on local disk — or, with an S3-compatible
backend, persists a **single node** to object storage you own.

Scaling that into a **distributed, multi-node vector database on object storage** — the tier that
serves billions of vectors from cheap storage behind a hot SSD/RAM cache — is **DartVault**, a
commercial product that runs on object storage *you* own. This page explains the divide, and the
economics behind it.

[NOTE]
====
It is the same `VectorCollection` API throughout. You start embedded and free, and grow into
DartVault without rewriting your application.
====

The library is the on-ramp; DartVault is the destination when a single node is no longer enough. The
rest of this page is about DartVault — what it is, why object storage, and how it compares — followed
by the one piece that ships free: single-node persistence to your own object store.

== Why object storage

Most vector databases keep the index in RAM. That makes queries fast and the bill large: RAM
costs roughly $2–5/GB-month, so a RAM-resident index runs on the order of **$3,600/TB-month**
once replication and SSD are included. Object storage costs about **$0.02/GB-month** — and NVMe
SSD, only ~4–5× slower than RAM, about $0.10/GB. Serving from object storage with an SSD/RAM
cache lands near **$70/TB-month**: 10–100× cheaper than keeping everything in memory.

This is not a bet. The market has already moved: a wave of serverless and open-source vector
databases has made object storage the durable tier specifically to escape RAM-bound economics.
What none of them offer is that model **managed for you on object storage you own, on the JVM** —
your vectors never leave your account. That is the gap DartVault fills.

== How DartVault works

Object storage is the source of truth; RAM and local SSD are caches in front of it.

[cols="1,3"]
|===
| Durable floor (object storage)
| Every write appends to a sequence-numbered write-ahead log under the collection's object-storage
  prefix, and committed generations are snapshotted as immutable objects. A write is durable once
  the object store acknowledges it — no synchronous multi-node quorum.

| Tiered cache
| A four-tier cascade (T0→T3) keeps 1-bit RaBitQ codes hot in JVM heap, warm clusters on local SSD,
  and full-precision vectors in object storage, promoting and evicting by access frequency. A cold
  namespace's first query fetches from object storage and warms the cache; subsequent queries are
  served locally.

| Centroid routing, not a graph
| Search routes through an IVF centroid hierarchy — global routing replicated to every node, then
  hierarchical refinement — that bounds the number of object-storage round-trips to the height of
  the tree. This is deliberate: graph indexes like HNSW require many sequential hops, which is fatal
  over high-latency object storage. It mirrors the centroid-tree approach the object-storage-first
  vector databases adopt for the same reason.

| Consistency
| New data is served by scanning the unindexed tail of the log until asynchronous indexing catches
  up, so a write is visible to the next query. Indexing never blocks reads or writes.
|===

The object-storage systems are proprietary-cloud or cluster-heavy; the embeddable ones are RAM-bound
and Python-centric. A managed, object-storage-backed engine that runs on storage you own — and whose
core also embeds free on the JVM — sits in the empty intersection.

== Honest trade-offs

Object storage trades latency for cost and scale. Warm queries — served from the SSD/RAM cache —
are fast. A **cold** query, against a namespace not yet cached, pays object-storage round-trips;
a leading object-storage vector database publishes roughly 874&nbsp;ms cold versus 14&nbsp;ms warm
p50 for a million vectors, and DartVault makes the same bargain. There is also a small
consistency-check floor on reads.

So DartVault is built for workloads that tolerate cold-start latency in exchange for cheap scale:
RAG and semantic search, batch and analytical retrieval, and large multi-tenant corpora where most
data is cold. For a small corpus on a strict single-digit-millisecond hot path, an in-memory store
is the right tool — the embedded `vectors` engine on local disk. DartVault does not try to win that
fight.

== Single-node object-storage persistence (free)

You do not need DartVault to put *one* node on object storage. The free library can point its durable
tier at an S3-compatible bucket — a single process, persisted to your own storage instead of local
disk. This is **not** the distributed database: there is no multi-node cache and no
generation-shipping across nodes. But if you just want one JVM backed by S3/R2, it ships in the
Apache library.

Any S3-compatible endpoint works — AWS S3, Cloudflare R2, Google Cloud Storage, or a local MinIO —
because the backend supports an endpoint override, path-style addressing, and static credentials:

[source,java]
----
StorageBackend objectStore = S3StorageBackend.create(
    URI.create("http://localhost:9000"), // endpoint (omit override for AWS S3)
    "vectors",                            // bucket
    "us-east-1",                          // region
    "minioadmin",                         // access key
    "minioadmin");                        // secret key

try (VectorCollection collection = VectorCollection.builder()
    .dimension(4)
    .metric(SimilarityFunction.COSINE)
    .indexType(IndexType.HNSW)
    .storagePath(Path.of("/var/lib/myapp/cache/demo")) // local cache
    .objectStore(objectStore, "demo/")                 // durable floor
    .build()) {

  collection.add(Document.of("a", new float[] {1, 0, 0, 0}, "alpha"));
  collection.commit();                                  // ships the generation to object storage
}
----

On `build()` the collection hydrates its local cache from the object store, so a fresh process picks
up the committed data automatically. When one node stops being enough, the same API grows into
DartVault.
